1. AI 요약 봇의 3단계 작동 원리 (Principles)
AI 봇이 채팅방에 개입하여 요약을 제공하는 과정은 크게 3가지 원리로 돌아갑니다.

① 트리거 (Trigger) & 데이터 수집 원리
봇이 언제, 어떤 데이터를 읽을 것인가를 결정합니다.

트리거 방식: 누군가 채팅창에 !요약 이라는 명령어를 쳤을 때 작동하게 하거나, 혹은 '새로운 사용자가 방에 처음 입장했을 때' 자동으로 작동하도록 서버의 이벤트 리스너가 감지합니다.

데이터 수집: 트리거가 발동되면, Spring Boot 서버가 MariaDB에 쿼리를 날려 가장 최근 대화 내역(예: 최근 50건 또는 최근 1시간 분량)의 sender와 content를 시간순으로 쭉 긁어옵니다.

② 프롬프트 가공 및 API 통신 원리 (LLM Processing)
수집된 날것의 데이터를 AI가 이해할 수 있게 포장해서 외부 뇌(OpenAI, Claude 등)로 보냅니다.

서버는 긁어온 채팅 텍스트를 하나의 긴 문자열로 합칩니다.

여기에 시스템 프롬프트를 씌웁니다. "너는 이 익명 채팅방의 친절한 요약 봇이야. 아래 대화 내역을 읽고, 사람들이 무슨 주제로 이야기하고 있었는지 3줄로 핵심만 요약해 줘. 말투는 발랄하게 해."

이 포장된 텍스트를 외부 AI API로 HTTP 전송합니다.

③ 가상 유저 브로드캐스팅 원리 (Broadcasting)
AI의 답변을 받아 사람들에게 전달합니다.

외부 AI 서버가 요약된 텍스트를 반환합니다.

우리 서버는 이 텍스트를 받아 MessageDTO 객체에 담습니다. 이때 sender의 이름은 "🤖 AI 요약봇"으로, messageType은 일반 채팅과 구분되는 "SYSTEM" 또는 "BOT"으로 설정합니다.

이 객체를 기존에 구축해 둔 웹소켓 파이프라인(/chat)을 통해 현재 접속 중인 모든 사람(또는 요청한 사람)에게 쏘아 보냅니다.


===================== 수정

말씀하신 대로 !요약 결과를 방 전체(Broadcast)에 쏴버리면, 한창 대화 중이던 기존 사람들은 갑자기 화면을 덮는 긴 요약 텍스트 때문에 대화의 흐름이 끊기고 짜증이 날 것입니다.

즉, 요약을 요청한 '그 사람에게만(Unicast)' 결과를 몰래 보내주는 것이 실무 메신저 설계의 정답입니다.

1. ChatGPT API는 계정 없이 무료로 쓸 수 있을까?
결론부터 말씀드리면 아닙니다. 반드시 계정(ID)이 필요하고, 쓴 만큼 비용이 발생하는 유료 서비스입니다.

여기서 실무자들이 반드시 구분해야 하는 개념이 있습니다. 바로 '웹 서비스'와 'API'의 차이입니다.

웹 서비스 (무료): 우리가 브라우저에서 chatgpt.com에 접속해서 타자를 치는 것은 무료입니다. 이건 사람이 직접 와서 광고도 보고 서비스도 써보라고 열어둔 '매장'입니다.

API (유료): 개발자가 자바 코드로 ChatGPT의 뇌(서버)에 직접 데이터를 꽂아 넣고 결과만 빼오는 것을 OpenAI API라고 합니다. 이건 기계가 초당 수십 번씩 요청을 보낼 수 있는 '도매상 전용 파이프라인'입니다.

계정과 키: 이 파이프라인을 뚫으려면 반드시 OpenAI 개발자 계정을 만들고, 신용카드를 등록한 뒤 API Key라는 비밀번호를 발급받아 스프링 부트 설정 파일에 넣어야 합니다.

비용 (토큰제): 보낸 글자 수와 받은 글자 수(Token)를 계산해서 0.1원 단위로 과금됩니다. 테스트용으로 혼자 몇 번 쏘는 건 한 달에 몇백 원 수준으로 아주 저렴하지만, 완전 무제한 무료는 존재하지 않습니다.

2. 1:1 채팅방은 유니캐스트(Unicast)로 구현하는 걸까?
이 질문은 정말 훌륭한 백엔드 아키텍처적 고민입니다! 절반은 맞고, 절반은 다릅니다.

① 귓속말 = 유니캐스트 (Unicast)
아까 말씀드린 요약 봇의 결과나, 단체방 안에서 A가 B에게 몰래 보내는 귓속말은 **유니캐스트(1:1)**가 맞습니다. 방 전체 세션 중 딱 B의 파이프만 찾아서 메시지를 쏙 넣어주는 원리입니다.

② 1:1 채팅방 = 멀티캐스트 (Multicast / Room ID)
하지만 카카오톡의 1:1 채팅방처럼 아예 독립된 방을 새로 파는 기능은, 유니캐스트라기보다는 **'특정 방 번호(Room ID)를 기준으로 하는 미니 브로드캐스트'**에 가깝습니다.

원리: A와 B가 1:1 대화를 시작하면 서버는 room_123이라는 새로운 가상의 방을 만듭니다.

구독: A와 B의 세션에 room_123이라는 꼬리표를 달아줍니다.

발송: A가 메시지를 보내면, 서버는 전체 유저가 아니라 room_123 꼬리표를 가진 세션들(A와 B)에게만 메시지를 뿌립니다. 즉, 2명만 들어있는 방에 브로드캐스트(멀티캐스트)를 하는 구조입니다.

어떤 무료 API를 쓸지 마음속으로 고르셨다면, 이제 백엔드 서버에서 이 API를 **'어떻게 호출할 것인가'**를 설계해야 하는 실무 고도화 단계가 남았습니다.

우리가 만든 채팅 서버는 지연 없이 빠르게 돌아가는 웹소켓(WebSocket) 기반입니다. 그런데 서버가 API에 요약을 요청할 때, 자바의 전통적인 방식인 RestTemplate이나 HttpURLConnection을 쓰면 AI가 대답을 줄 때까지 서버의 스레드가 멈춰서 기다리는(Blocking) 치명적인 문제가 발생합니다.

실무 해결책 (Non-Blocking): 스프링 부트에서 제공하는 최신 비동기 통신 객체인 **WebClient**를 도입해야 합니다.

작동 원리: 서버가 API에 질문을 던지고 나서 "답변 오면 나중에 알려줘!" 하고 바로 돌아와 다른 사람들의 채팅을 계속 처리합니다. AI의 답변이 도착하면 그때 스레드가 다시 깨어나서 사람들에게 결과를 뿌려줍니다.

가장 추천해 드리는 Gemini API나 Groq API 중 하나를 타겟으로 잡고, 이 똑똑하고 멈춤 없는 WebClient를 스프링 부트에 설정하는 코드 작업부터 바로 시작해 볼까요? 어떤 API가 더 마음에 드시나요?
