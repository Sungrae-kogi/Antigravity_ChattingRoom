# JWT(JSON Web Token) 도입 계획서

현재의 세션(Session) 기반 인증에서 토큰 기반 인증인 JWT로 마이그레이션하기 위한 계획입니다. 

## 1. 이론: 세션과 JWT의 차이점

### 기존 세션 방식 (Stateful)
- **방식**: 서버가 사용자의 로그인 상태(`JSESSIONID`)를 메모리나 DB에 기억합니다.
- **장점**: 구현이 쉽고, 서버가 사용자 상태를 통제하기 쉽습니다 (예: 강제 로그아웃).
- **단점**: 사용자가 많아지면 서버 메모리를 많이 차지합니다. 서버가 여러 대(Scale-Out)일 경우 세션 정보를 공유해야 하는 복잡성(Redis Session 등)이 발생합니다.

### JWT 방식 (Stateless)
- **방식**: 서버는 아무것도 기억하지 않습니다! 대신 로그인 성공 시, 유저 정보와 암호화된 서명이 담긴 **'출입증(JWT)'**을 발급해 클라이언트에게 줍니다. 클라이언트는 이후 요청마다 이 출입증을 보여줍니다.
- **구조**: `Header(규격)` + `Payload(유저 정보)` + `Signature(서버만 아는 비밀키로 만든 서명)`
- **장점**: 서버가 상태를 저장하지 않아 가볍고, 서버를 여러 대 늘리기 쉽습니다(확장성). 모바일 앱이나 다른 플랫폼과 연동하기도 좋습니다.
- **단점**: 한 번 발급된 토큰은 뺏거나 무효화하기 어렵습니다 (이를 보완하기 위해 발급 시간을 짧게 하고 Refresh Token을 사용합니다).

## 2. 동작 방식 (Operation Flow)

### A. 로그인 흐름
1. **사용자**: 아이디와 비밀번호를 서버(`/signin`)로 전송합니다.
2. **서버**: 아이디/비밀번호가 맞는지 DB와 대조합니다.
3. **서버 (JWT 발급)**: 인증에 성공하면 사용자의 ID(username) 등을 담아 JWT 토큰을 생성합니다.
4. **서버 -> 클라이언트**: 생성된 JWT를 응답 헤더나 바디에 담아 클라이언트로 보냅니다.
5. **클라이언트**: 받은 JWT를 브라우저(LocalStorage 또는 웹 쿠키)에 소중히 보관합니다.

### B. 일반 API 요청 흐름 (채팅방 입장 등)
1. **클라이언트**: 서버에 요청을 보낼 때, HTTP 헤더(`Authorization: Bearer <token>`)에 자신이 보관하던 JWT를 껴서 보냅니다.
2. **서버 (JwtFilter)**: 요청이 컨트롤러에 닿기 전에 필터가 가로채서 헤더의 토큰을 확인합니다.
3. **서버 (검증)**: 서버가 가진 비밀키로 서명이 위조되지 않았는지, 유효기간이 안 지났는지 검사합니다.
4. **결과**: 합격이면 통과(SecurityContext에 인증 정보 세팅), 불합격이면 401(Unauthorized) 에러를 반환합니다.

### C. 웹소켓(WebSocket) 통신 흐름 (가장 큰 관건)
웹소켓 최초 연결(Handshake) 시에는 일반적인 HTTP 방식과 달라서 Authorization 헤더를 넣기가 곤란합니다.
- **해결책**: 클라이언트가 웹소켓 연결을 요청할 때 URL 파리미터로 토큰을 보내게 합니다. (예: `ws://localhost:8080/ws/chat?token=여기에JWT`)
- **서버 검증**: `WebSocketInterceptor` 또는 `ChatHandler` 연결 시점에 파라미터에서 토큰을 추출해 검증하고 세션에 유저 정보를 세팅합니다.

## 3. 단계별 구현 순서 (Implementation Sequence)

> [!IMPORTANT]
> 아래 순서대로 차근차근 진행할 예정입니다. 변경 범위가 넓으므로 한 단계씩 검증하며 넘어가는 것이 중요합니다.

### Step 1: 의존성 추가 및 비밀키 설정
- `pom.xml`에 JWT 관련 라이브러리(jjwt 등)를 추가합니다.
- `application.properties`에 토큰 암호화에 쓸 기밀 키(Secret Key)와 만료 시간을 설정합니다.

### Step 2: JWT 도구 클래스 생성 (`JwtUtil`)
- 토큰을 생성하고, 토큰에서 유저명을 꺼내고, 토큰이 유효한지 검사하는 자판기 역할의 유틸리티 클래스를 만듭니다.

### Step 3: JWT 검증 필터 생성 (`JwtAuthenticationFilter`)
- 모든 HTTP 요청을 맨 앞에서 검사할 문지기(Filter)를 만듭니다. 
- 헤더에서 토큰을 빼내 `JwtUtil`에게 검사를 맡기고, 통과하면 Spring Security에 "이 사람은 인증됐어"라고 알려주는 역할을 합니다.

### Step 4: Spring Security 설정 변경 (`SecurityConfig`)
- 이제 세션을 쓰지 않겠다고 선언합니다. (`sessionCreationPolicy(SessionCreationPolicy.STATELESS)`)
- 기본 폼 로그인 로직들을 비활성화하고, 우리가 만든 `JwtAuthenticationFilter`를 필터 체인에 끼워 넣습니다.

### Step 5: 인증 컨트롤러 개편 (`AuthController`)
- 기존의 세션 기반 로그인이 폼 로그인 컨트롤러 대신, `/api/auth/login` 같은 REST API를 만들어 직접 아이디/비번을 검증하고 성공 시 JWT를 JSON 또는 쿠키로 반환하도록 수정합니다.

### Step 6: 프론트엔드 연동 변경 (`signin.jsp` 등)
- 로그인 시 폼 서브밋(POST) 대신 JavaScript `fetch`나 `axios`로 요청을 보냅니다.
- 받아온 JWT를 로컬 스토리지에 저장하고, 이후 서버에 요청을 보낼 때마다 헤더에 토큰을 실어 보냅니다.

### Step 7: 웹소켓 인증 적용 (`ChatHandler` & 프론트 JS)
- `chat.js`에서 `new WebSocket()` 호출 시 URL 뒤에 JWT를 붙여서 요청합니다.
- 서버 측 웹소켓 인터셉터나 핸들러에서 이 토큰을 꺼내 검증하고 사용자를 식별하게 바꿉니다.
